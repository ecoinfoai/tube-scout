# Feature Specification: Takeout 통합 적재와 운영 효율화

**Feature Branch**: `017-takeout-unified-ingest`
**Created**: 2026-05-16
**Status**: Draft
**Input**: User description: "Takeout 통합 적재와 운영 효율화 — ffprobe 메모이즈 + 적재·자막·지문 통합 명령 + 영상 본체 삭제 정책"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 한 학과 archive 를 한 명령으로 끝까지 처리 (Priority: P1)

학과 운영자가 새로 풀린 Takeout archive 한 묶음을 본 시스템에 적재할 때, 단일 명령 한 번으로 메타데이터 적재·자막 생성·지문 추출까지 모두 마치고 결과를 한 번에 확인한다. 운영자는 단계별 명령 순서를 기억하거나 중간 산출물 위치를 추적할 필요가 없다.

**Why this priority**: 사용자 의도 6 단계 흐름 (정합성 확인 → DB 저장 → 음원 추출 → 자막 + 지문 → 별도 저장 + DB 참조 → 분석) 의 핵심 운영 표면이다. 현재 4–5 회 분리 호출이 운영자의 일상 인지 부담을 만들고 있어 22 학과 확장 시점에 운영 시간을 정확히 추산할 수 없다.

**Independent Test**: 회귀 환경에서 비어 있는 작업 디렉토리 + 간호학과 sample archive 만 주고 통합 명령 한 번을 실행하여, 명령 종료 시점에 영상 메타데이터, 자막 파일 9 개, 지문 데이터 9 개가 모두 만들어져 있음을 확인한다.

**Acceptance Scenarios**:

1. **Given** 학과 alias 가 등록되어 있고 Takeout archive 한 묶음이 풀려 있는 상태, **When** 통합 명령 한 번을 실행, **Then** 영상 메타데이터·자막·지문이 모두 만들어지고 운영자에게 한 묶음 단위의 처리 요약 (적재 영상 수, 자막 성공/실패 수, 지문 성공/실패 수, 소요 시간) 이 표시된다.
2. **Given** archive 안에 mp4 본체가 동봉되지 않은 영상이 다수 포함된 상태, **When** 통합 명령 실행, **Then** mp4 부재 영상은 메타데이터만 저장되고 자막·지문 단계는 자동 skip 되며 그 사유가 감사 로그에 기록된다.
3. **Given** 같은 archive 가 두 번째로 적재되는 상태, **When** 통합 명령 실행, **Then** 데이터베이스 행 수와 자막·지문 파일이 첫 번째 실행 결과와 동일하게 유지되고, 운영자에게 "변경 0" 으로 표시된다.

### User Story 2 - 적재 시간을 사람이 기다릴 수 있는 수준으로 단축 (Priority: P1)

학과 운영자가 통합 명령을 호출했을 때, 처리 시간이 사람이 화면을 보며 기다릴 수 있는 수준 안에서 끝난다. 현재 한 학과 archive 처리에 약 17 분이 걸리는 적재 단계가 분 단위로 줄어든다.

**Why this priority**: 22 학과 확장 시점에 운영자가 종일 명령 종료를 기다리는 상황을 막아야 한다. 적재 시간 단축은 통합 흐름(US1) 의 가치를 실제로 실현하는 전제 조건이며, 통합 흐름이 도입되어도 한 단계가 17 분이면 총 소요 시간이 변하지 않는다.

**Independent Test**: 간호학과 sample archive (mp4 9 개 + 메타 2554 행) 의 적재 단계만 분리 측정하여 1 분 이내에 완료됨을 확인한다. 적재 단계 산출물 (영상 메타데이터 행 수, 매핑 confidence 분포) 은 spec 016 의 결과와 일치해야 한다.

**Acceptance Scenarios**:

1. **Given** 간호학과 sample archive 가 풀려 있는 상태, **When** 적재 단계 단독 측정, **Then** 본 작업 머신에서 1 분 이내에 완료된다.
2. **Given** 한 archive 안에 mp4 9 개가 들어 있는 상태, **When** 적재 단계 실행, **Then** mp4 본체에서 길이를 측정하는 외부 도구 호출 횟수가 mp4 개수와 같다 (영상 후보 행 수만큼 반복되지 않는다).
3. **Given** 같은 archive 적재를 두 번째로 실행, **When** 두 번째 적재, **Then** 산출물과 매핑 결과는 첫 번째와 동일하고, 두 번째 적재의 소요 시간도 첫 번째와 같은 분 단위 안에 들어온다.

### User Story 3 - 영상 본체 삭제 전 운영자 확인 (Priority: P2)

학과 운영자가 통합 명령으로 한 묶음 처리를 마쳤을 때, 시스템이 어떤 영상 파일을 삭제하려고 하는지 명시적으로 표시하고 yes/no 확인을 받은 뒤에야 실제 삭제를 수행한다. 학과 영상의 원본은 Google Drive 같은 별도 보관장치에 있으므로 사본 삭제는 안전하나, 잘못된 파일을 지우는 사고를 방지한다.

**Why this priority**: 사용자 의도 (분석 직후 영상 삭제, 학과 자기관리 책임 위임) 에 직접 부합하지만, US1 + US2 가 먼저 가능해야 운영자가 통합 흐름의 마지막 단계로 영상 삭제를 자연스럽게 경험할 수 있다. 영상 삭제 자체는 운영 흐름의 종결 단계라 P2 가 적절하다.

**Independent Test**: 통합 명령이 모든 분석 단계를 마친 직후, 삭제 후보 영상 파일 목록과 총 디스크 용량을 표시하고, 운영자가 거부 (no) 했을 때 영상이 그대로 남아 있는지, 승인 (yes) 했을 때 archive 의 mp4 본체와 작업 디렉토리의 심볼릭 링크가 모두 정리되고 감사 로그에 기록되는지 두 경우 모두 확인한다.

**Acceptance Scenarios**:

1. **Given** 통합 명령의 분석 단계가 모두 끝난 상태, **When** 시스템이 영상 삭제 단계에 진입, **Then** 운영자에게 삭제 후보 파일 목록과 총 용량이 표시되고 yes/no 응답을 기다린다.
2. **Given** 운영자가 yes 응답을 한 상태, **When** 시스템이 삭제 수행, **Then** archive 의 mp4 본체와 작업 디렉토리의 심볼릭 링크가 모두 정리되고, 삭제된 파일 수와 회수된 용량이 표시되며 감사 로그에 기록된다.
3. **Given** 운영자가 no 응답을 한 상태, **When** 시스템이 삭제 거부를 받음, **Then** 영상 파일이 그대로 남아 있고, 거부 사실이 감사 로그에 기록되어 후속 운영 추적이 가능하다.
4. **Given** 자막 또는 지문 단계가 일부 영상에서 처리 실패한 상태, **When** 시스템이 영상 삭제 단계에 진입, **Then** 운영자에게 처리 실패 영상 목록을 표로 먼저 표시하고, 그 다음 화면에서 처리 성공한 영상만 삭제 후보로 한 번 더 yes/no 확인을 받는다. 처리 실패 영상은 삭제 후보에서 자동 제외되고 다음 통합 명령 호출이 우선 재시도할 수 있도록 별도의 재시도 매니페스트 파일에 정리된다.

### User Story 4 - 22 학과 확장을 위한 운영 baseline (Priority: P3)

시스템 관리자가 통합 명령의 단계별 소요 시간과 결과 분포를 학과별로 비교 가능한 형식으로 확인하여, 22 학과 확장 시점에 운영 자원 (작업 시간, 디스크) 을 계획할 수 있다.

**Why this priority**: US1 + US2 + US3 의 사용 경험이 충분히 누적되면 자연스럽게 측정값이 모이지만, 처음부터 명령이 측정값을 출력 형식에 포함하면 baseline 수립이 수월하다. 기능적으로는 보조이므로 P3.

**Independent Test**: 통합 명령의 종료 시점 출력에 적재·자막·지문·삭제 4 단계의 소요 시간이 모두 양의 값으로 표시되고, 같은 archive 를 두 번 실행해도 분 단위 측정값이 안정적으로 재현되는지 확인한다.

**Acceptance Scenarios**:

1. **Given** 통합 명령이 정상 완료된 상태, **When** 운영자가 종료 화면을 확인, **Then** 단계별 (적재·자막·지문·삭제) 소요 시간과 성공/실패 카운트가 한 표 안에 표시된다.
2. **Given** 같은 archive 를 두 번 적재한 상태, **When** 두 번의 측정값을 비교, **Then** 적재·자막·지문 소요 시간이 분 단위로 동일한 범위에 들어온다.

---

### Edge Cases

- 통합 명령 중간 단계 (자막 또는 지문) 가 부분 실패한 경우 — 실패한 영상의 처리 결과는 감사 로그에 기록되고, 영상 삭제 단계는 US3 의 부분 실패 정책에 따른다 (위 미결 사항).
- 통합 명령 실행 중 사용자가 중단 (Ctrl+C) 한 경우 — 적재까지 완료되었으면 데이터베이스 상태는 유지되고, 자막·지문 부분 산출물은 그대로 남아 다음 실행에서 멱등하게 이어진다. 영상 삭제 단계는 사용자 확인 전이라 실행되지 않는다.
- 영상 삭제 단계에서 archive 의 mp4 가 이미 다른 process 에 의해 점유된 경우 — 삭제 실패가 감사 로그에 기록되고, 나머지 파일은 그대로 삭제를 시도한다.
- 다른 학과의 적재가 동시에 진행 중인 경우 — 두 학과의 데이터 격리는 alias 별 작업 디렉토리로 보장되며, 삭제 단계도 현재 처리 중인 학과의 파일만 대상으로 한다.
- mp4 본체와 메타데이터의 길이 정보가 일치하지 않는 경우 — spec 016 의 evidence-score 매핑 정책 (high / medium / ambiguous / unmapped) 이 그대로 적용되며, 통합 명령의 처리 흐름에 변화 없음.

## Requirements *(mandatory)*

### Functional Requirements

#### 적재 효율화 (US2)

- **FR-001**: 시스템은 한 archive 안의 mp4 파일 1 개에 대해 외부 도구로 영상 길이를 측정하는 호출을 최대 1 회만 수행해야 한다 (영상 메타 후보 행 수와 무관).
- **FR-002**: 시스템은 영상 메타와 mp4 본체의 매핑 결정 단계에서 mp4 1 개의 길이 정보를 캐싱하여 같은 mp4 에 대한 반복 측정을 차단해야 한다.
- **FR-003**: 시스템은 적재 단계의 소요 시간을 명령 종료 후 측정값으로 출력해야 한다 (spec 016 의 FR-022 보존).
- **FR-004**: 시스템은 spec 016 의 적재 결과 정합성 (영상 메타 행 수, privacy 값 표준화, 매핑 confidence 분포) 을 효율화 후에도 동일하게 유지해야 한다.

#### 통합 명령 흐름 (US1)

- **FR-005**: 시스템은 한 학과 archive 한 묶음의 처리를 단일 명령 한 번으로 완료할 수 있는 호출 표면을 제공해야 한다.
- **FR-006**: 시스템은 통합 명령 안에서 적재 → 음원 추출 → 자막 생성 → 지문 추출 → 영상 삭제 (US3) 5 단계를 순차로 수행하며, 각 단계의 완료를 운영자에게 실시간으로 표시해야 한다.
- **FR-007**: 시스템은 통합 명령 안에서 한 영상의 음원을 한 번만 추출하고, 추출된 음원을 자막 단계와 지문 단계가 함께 사용한 뒤 분석 완료 즉시 정리해야 한다 (spec 013 의 임시 음원 즉시 삭제 정책 보존).
- **FR-008**: 시스템은 mp4 본체가 동봉되지 않은 영상에 대해 자막·지문 단계를 자동 skip 하고 감사 로그에 사유를 기록해야 한다 (spec 016 의 FR-008 / FR-019 보존).
- **FR-009**: 시스템은 같은 archive 가 두 번째로 통합 명령으로 처리될 때 데이터베이스 행 수가 변하지 않아야 하며, 이미 생성된 자막·지문 파일은 재생성하지 않고 재사용해야 한다 (spec 016 의 FR-009 멱등성 보존).
- **FR-010**: 시스템은 통합 흐름을 별도의 신규 명령 (예: `collect ingest` 또는 동등한 새 이름) 으로 노출한다. 기존 분리 명령 (`collect takeout`, `collect audio-extract`, `collect process-audio`, `collect transcripts`, `collect fingerprint`) 의 호출 표면과 기본 동작은 변경하지 않고 backward compatibility 를 유지한다. quickstart 의 운영자 권장 흐름은 신규 통합 명령으로 일원화된다.

#### 영상 본체 삭제 (US3)

- **FR-011**: 시스템은 운영자가 통합 명령에 `--delete-source` 옵션 (또는 동등한 명시 옵션) 을 지정한 경우에만 영상 삭제 단계에 진입한다. 옵션이 지정되지 않으면 영상은 모두 보존되고 어떤 prompt 도 등장하지 않는다.
- **FR-012**: 시스템은 `--delete-source` 옵션이 지정된 상태에서 영상 삭제 단계에 진입할 때, 먼저 처리 실패 영상 목록 (자막 또는 지문 단계가 실패한 영상의 video_id, 영상 제목, 실패 단계, 실패 사유) 을 표로 운영자에게 표시한 뒤, 두 번째 화면에서 처리 성공한 영상만 삭제 후보로 한 번 더 yes/no 확인을 받는다. 즉 두 단계 prompt 구조로 처리 실패 영상이 운영자에게 명확히 인지되도록 한다. **두 번째 prompt 의 timeout 정책**: 별도의 시간 제한을 두지 않고 운영자 응답을 무한 대기한다 (Rich `Confirm.ask` default). "응답 없음" 분기는 EOF / Ctrl+D 입력으로만 발생하며, Ctrl+C 는 별도의 `interrupted` 분기로 처리한다.
- **FR-013**: 시스템은 두 번째 prompt 에서 운영자의 yes 응답 시 처리 성공 영상의 archive 안 mp4 본체와 작업 디렉토리의 심볼릭 링크를 정리하고, 정리한 파일 수와 회수된 용량을 운영자에게 표시하며 감사 로그에 기록해야 한다. 처리 실패 영상은 삭제 후보에서 자동 제외된다.
- **FR-014**: 시스템은 두 번째 prompt 에서 운영자의 no 응답이거나 응답이 없는 경우 (timeout 또는 중단) 모든 영상 파일을 그대로 두고, 거부 사실을 감사 로그에 기록한다.
- **FR-015**: 시스템은 처리 실패 영상 (자막 또는 지문 단계 실패) 의 목록을 별도의 재시도 매니페스트 파일 (예: `data/<alias>/retry_pending.json` 또는 동등한 위치) 로 자동 정리하고, 다음 통합 명령 호출 시 이 매니페스트의 영상이 우선 재시도 대상으로 선택되도록 한다. 매니페스트의 row 는 video_id, 마지막 시도 시각, 실패 단계, 실패 사유를 포함한다.

#### 운영 baseline (US4)

- **FR-016**: 시스템은 통합 명령 종료 시 단계별 (적재·자막·지문·삭제) 소요 시간과 성공/실패 카운트를 한 표로 운영자에게 표시해야 한다.
- **FR-017**: 시스템은 각 단계의 측정값을 감사 로그의 row 에 보존하여 학과별·시간대별 baseline 비교가 가능하게 해야 한다 (spec 016 의 FR-023 보존).
- **FR-018**: 시스템은 재시도 매니페스트의 row 가 성공적으로 처리된 시점에 해당 row 를 매니페스트에서 제거하거나 상태를 `resolved` 로 표시한다. 매니페스트는 빈 상태로 남아 있으면 다음 명령 호출에 영향을 주지 않는다.

### Key Entities

- **통합 처리 결과 요약**: 한 학과 archive 한 묶음을 통합 명령으로 처리한 결과의 단일 출력 단위. 적재 결과 (영상 메타 행 수, 매핑 confidence 분포, mp4 매핑 수), 자막 생성 결과 (성공/실패 수, 소요 시간), 지문 추출 결과 (성공/실패 수, 소요 시간), 영상 삭제 결과 (삭제 수, 회수 용량, 운영자 응답), 재시도 매니페스트 갱신 결과 (신규 추가 수, 해소 수) 5 단위를 포함한다.
- **처리 실패 영상 표**: 영상 삭제 단계의 첫 번째 prompt 에서 운영자에게 표시되는 표. 자막 또는 지문 단계가 실패한 영상의 video_id, 영상 제목, 실패 단계, 실패 사유를 행으로 가지며, 운영자가 부분 실패 영상의 존재를 명확히 인지하도록 한다.
- **삭제 후보 영상 목록**: 영상 삭제 단계의 두 번째 prompt 에서 운영자에게 표시되는 파일 목록. 모든 분석 단계가 성공한 영상에 한정되며, archive 안의 mp4 경로, 작업 디렉토리의 심볼릭 링크 경로, 파일별 용량을 포함한다.
- **운영자 응답 기록**: 두 prompt 각각에서 운영자가 yes 또는 no 로 응답한 사실을 감사 로그에 기록하는 row. 응답 시각, 응답 내용, 대상 영상 수, 학과 alias 를 포함한다.
- **재시도 매니페스트**: 처리 실패 영상의 video_id 와 실패 단계·사유·마지막 시도 시각을 보존하는 별도 파일 (예: `data/<alias>/retry_pending.json` 또는 동등한 위치). 다음 통합 명령 호출 시 우선 재시도 대상이 된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 본 작업 머신에서 간호학과 sample archive (mp4 9 개 + 메타 2554 행) 의 적재 단계가 1 분 이내에 완료된다. spec 016 시점의 약 17 분 대비 약 17 배 개선.
- **SC-002**: 한 학과 archive 한 묶음 처리에 필요한 운영자의 명령 호출 횟수가 1 회로 줄어든다. spec 016 시점의 4–5 회 대비 80% 이상 단축.
- **SC-003**: 영상 본체 삭제 단계에서 운영자의 명시적 yes 응답 없이 파일이 삭제되는 경우가 0 건이다. 모든 삭제 사례가 감사 로그에 응답 사실과 함께 기록된다.
- **SC-004**: 같은 archive 를 두 번 통합 명령으로 처리해도 데이터베이스의 영상 메타 행 수와 자막·지문 파일 수가 첫 번째 처리 결과와 동일하게 유지된다 (멱등성 회귀).
- **SC-005**: 한 번의 통합 명령 안에서 한 영상에 대한 음원 디코딩이 1 회로 제한된다 (자막 단계와 지문 단계가 같은 임시 음원을 공유).
- **SC-006**: 통합 명령 종료 시 운영자에게 표시되는 처리 요약에 단계별 (적재·자막·지문·삭제) 소요 시간이 모두 양의 측정값으로 표시된다.
- **SC-007**: 처리 실패 영상 (자막 또는 지문 단계 실패) 은 영상 삭제 단계의 첫 번째 prompt 에서 100% 운영자에게 표로 노출되고, 두 번째 prompt 의 삭제 후보 목록에는 0 건이 포함된다. 즉 처리 실패 영상이 의도치 않게 삭제되는 경우가 0 건이다.
- **SC-008**: 처리 실패 영상이 재시도 매니페스트에 자동 정리되어, 다음 통합 명령 호출 시 100% 우선 재시도 대상으로 잡힌다. 매니페스트에 누락되는 실패 영상이 0 건이다.

## Assumptions

- **영상 재분석 흐름**: 자막 모델 업그레이드 등으로 같은 영상을 다시 처리해야 할 때, 운영자는 Google Drive 같은 별도 보관장치에서 영상을 다시 내려받아 archive 를 재생성한 뒤 통합 명령을 다시 호출한다. 본 시스템은 영상 본체의 영구 보관 책임을 지지 않는다.
- **버전 등급**: 본 spec 이 도입하는 통합 명령 흐름과 영상 삭제 정책은 운영자의 호출 패턴을 변경하므로, release 시점에 MINOR 버전 (예: 0.6.0) 으로 bump 하는 것을 권장한다. PATCH (0.5.2) 로 진행할지 MINOR (0.6.0) 으로 진행할지는 plan 단계의 첫 task 에서 commit 한다.
- **기존 분리 명령의 backward compatibility**: spec 013 / spec 016 의 기존 분리 명령은 통합 명령 도입 후에도 호출 가능한 상태로 유지된다. 단 quickstart 의 권장 흐름은 통합 명령으로 일원화된다. 통합 명령은 별도의 신규 명령 (예: `collect ingest`) 으로 노출되며 기존 명령의 호출 표면은 변경되지 않는다 (Q1=C 결정).
- **영상 삭제 옵트인 정책**: 통합 명령 호출 시 운영자가 `--delete-source` 옵션 (또는 동등한 명시 옵션) 을 지정하지 않으면 영상 삭제 단계 자체가 실행되지 않는다 (Q2=B 결정). 옵션을 지정한 경우에만 두 단계 prompt 가 등장한다.
- **처리 실패 영상의 자동 보존**: 자막 또는 지문 단계가 실패한 영상은 영상 삭제 후보에서 자동으로 제외되며, 별도의 재시도 매니페스트로 정리되어 다음 통합 명령 호출에서 우선 처리된다 (Q3 사용자 정의 결정).
- **임시 음원의 비영구화**: 통합 명령 안에서 추출된 임시 음원 파일은 분석 완료 즉시 정리되며 디스크에 영구 보관되지 않는다 (spec 013 의 정책 보존).
- **사용자 메모리 정책 보존**:
  - 사용자 메모리 `project_takeout_pivot_20260512` 의 yt-dlp 영구 폐기 결정 보존 (자막 source 는 ASR 단일 경로).
  - 사용자 메모리 `project_scope_decisions_20260506` 의 OCR · 화자분리 영구 제외 결정 보존 (영상 보관의 추가 분석 가치 없음).
  - 사용자 메모리 `feedback_response_style` 의 한글 풀어쓰기 정책을 운영자 노출 메시지에 적용.
- **작업 격리**: 본 spec 의 변경은 alias 별 작업 디렉토리 (`data/<alias>/`) 안에서 격리되며, 다른 학과 데이터에 영향을 주지 않는다.
- **감사 로그의 append-only 보장**: spec 013 의 감사 로그 append-only 특성을 유지하며, 영상 삭제 단계에서 추가되는 row 도 새 row append 로만 기록된다.

## Cross-Spec Boundaries

본 spec 의 변경은 다음 spec 의 기존 보장을 보존한다.

- **spec 013** (Takeout local ASR + reuse detection): 임시 음원 즉시 삭제 정책, 음원-자막-지문 흐름의 임시 파일 lifecycle, faster-whisper ASR 단일 경로, 감사 로그 append-only.
- **spec 016** (Takeout 적재 결함 PATCH): FR-009 멱등성, FR-008 / FR-019 mp4 부재 영상 skip, FR-022 적재 결과 카운트 출력, FR-023 감사 row 컬럼 셋, 영상 메타 2554 행 + privacy 표준화 + 한국어 quoting 안전 파싱.

본 spec 은 새 entity 를 도입하지 않고 기존 entity (감사 로그 row, 적재 결과 단위, 처리 상태 단위) 에 단계별 시간/응답 필드를 추가하는 방식으로 처리한다. 상세 데이터 구조는 plan 단계에서 결정.
