## Clarifications

### Session 2026-04-28

- Q: 운영자(신규 학과 등록·시크릿 매핑·OAuth 동의)의 인터페이스 형식은? → A: CLI 명령만 (예: `tube-scout admin add-department ...`). 운영자 GUI는 추가하지 않음 — "얇은 웹 레이어" 원칙·agenix 흐름·운영자 빈도 모두를 충족.
- Q: 세션·로그인 보호 수준은? → A: 표준 보호 — 8시간 세션 자동 만료(미활동 시 단축 가능), HTTPS 필수(리버스 프록시 종단), 로그인 5회 연속 실패 시 5분 잠금. 2FA·IP 화이트리스트는 본 idea 범위 외.
- Q: OAuth 토큰 만료 등 운영 알림은 어디로 보내는가? → A: CLI status 명령(`tube-scout admin status`) + 구조화 로그(파일 또는 journald). 만료 감지 즉시 로그에 기록되고 status 명령으로 즉시 조회 가능. 이메일·Slack은 본 idea 범위 외.
- Q: Analysis Job ID 형식과 결과 디렉터리 매핑은? → A: Job ID = `YYYYMMDD-HHMMSS` (기존 CLI projects/ 구조와 1:1), sub-second 충돌 시 `-2`, `-3` 카운터 접미사. 결과 디렉터리명은 Job ID와 동일.
- Q: 실패·중단된 작업의 부분 결과를 사용자에게 노출하는가? → A: 노출하지 않음. 대신 spec 007 FR-004의 checkpoint 메커니즘을 활용해 [재실행] 시 마지막 성공 단계 다음부터 자동으로 이어 실행한다. 부분 산출물 디스크 파일은 보존되나 UI 다운로드 링크는 제공하지 않는다.

## User Scenarios & Testing *(mandatory)*

### User Story 1 - 교무과 담당자가 학과·교수·과목·기간을 입력해 풀 분석 보고서를 받는다 (Priority: P1)

부산보건대 교무과 담당자(이하 사용자)가 사내망 웹 주소에 접속해 지정된 아이디·비밀번호로 로그인한다. 첫 화면에서 학과(드롭다운 또는 직접 입력) → 교수명 → 과목명 → 기간(영상 게시일 시작·종료)을 입력하고 [분석 시작]을 누르면 시스템이 해당 학과의 본 채널 OAuth 자격으로 영상 목록 수집 → 메타데이터 수집 → 자막 수집(공개·비공개 모두) → retention/analytics 수집 → 자막 기반 재사용 탐지 → 보고서 생성을 자동 실행한다. 진행 중에는 단계와 진행 카운트가 화면에 표시되고, 완료 시 같은 화면에서 HTML/PDF/Excel 보고서를 다운로드 또는 새 탭으로 열 수 있다.

**Why this priority**: 본 idea의 존재 이유. 사용자가 터미널·환경변수·OAuth 토큰·채널별 API 키를 직접 다루지 않고 4개 입력만으로 풀 파이프라인 결과를 받지 못하면 idea 가치는 0이다. 이 단일 흐름만으로도 "개발자 경유 없이 학과별 보고서 즉시 발행"이라는 핵심 효익이 달성되므로 단독 MVP가 성립한다.

**Independent Test**: 운영자가 학과 1개를 등록(agenix에 OAuth/API 시크릿 + 학과 메타데이터 매핑 추가)한 상태에서, 사용자가 웹 URL을 처음 받아 (1) 로그인 → (2) 학과·교수·과목·기간 입력 → (3) [시작] 클릭 → (4) 진행률 갱신 확인 → (5) 보고서 3종(HTML/PDF/Excel) 다운로드까지 터미널·파일시스템 접근 0회로 완료할 수 있으면 통과.

**Acceptance Scenarios**:

1. **Given** 운영자가 `physiology` 학과의 OAuth 시크릿·API 키를 agenix에 등록했고 학과↔alias 매핑이 시스템에 반영되었다, **When** 사용자가 로그인 후 학과 드롭다운에서 `물리치료과`를 선택하고 교수명 `홍길동`, 과목명 `생리학`, 기간 `2024-03-01 ~ 2025-02-28`을 입력하고 [분석 시작]을 누른다, **Then** 분석 작업이 큐에 등록되고 진행률 화면으로 전환된다.
2. **Given** 분석 작업이 실행 중이다, **When** 사용자가 진행률 화면을 본다, **Then** 현재 단계(영상 목록 → 메타데이터 → 자막 → retention → analytics → 재사용 탐지 → 보고서)와 단계별 처리 카운트(예: 자막 12 / 47 영상)가 5초 이내 간격으로 갱신된다.
3. **Given** 분석 작업이 정상 완료되었다, **When** 사용자가 결과 화면을 본다, **Then** v1~v3 영상 필터 보고서, retention/analytics 보고서, 재사용 탐지 보고서(idea4)에 대한 HTML·PDF·Excel 다운로드 링크가 같은 화면에 모두 노출된다.
4. **Given** 분석 작업 중 OAuth 토큰 만료 또는 API 쿼터 초과가 발생했다, **When** 사용자가 결과 화면을 본다, **Then** 한국어 오류 메시지(예: "OAuth 토큰 갱신이 필요합니다 — 운영자에게 문의하세요" 또는 "오늘 API 쿼터가 초과되었습니다 — 내일 다시 시도하세요")가 표시되고, agenix 키 이름·환경변수명·내부 경로는 노출되지 않는다.
5. **Given** 입력한 기간에 해당 학과·교수·과목으로 매칭되는 영상이 0건이다, **When** 분석이 진행된다, **Then** 시스템은 빈 보고서 대신 "조건에 맞는 영상이 없습니다 — 입력값을 확인하세요" 화면을 노출한다.

### User Story 2 - 과거 분석 이력을 같은 화면에서 다시 연다 (Priority: P2)

사용자가 학과장 보고나 누적 추적을 위해 과거 분석 결과를 다시 열어야 한다. 이력 화면에는 분석 이력 목록(학과 · 교수명 · 과목명 · 기간 · 생성일시 · 상태)이 최신순으로 보이고, 각 행을 클릭하면 해당 분석의 결과 화면으로 이동해 보고서를 다시 다운로드하거나 열람할 수 있다. 재사용 탐지 결과의 리뷰 상태(미검토/중복확정/오탐)도 이력에서 확인 가능하다.

**Why this priority**: 동일 분석을 매번 다시 돌리는 것은 시간(수십 분) + YouTube API 쿼터 낭비다. 단, P1이 동작해야 이력이 쌓이므로 P1 다음에 와야 한다. 이력 화면 없이도 P1만으로 신규 분석은 가능하므로 분리 가능한 단위.

**Independent Test**: P1을 통해 분석 이력 2건 이상이 쌓인 상태에서, 사용자가 이력 탭에 들어가 항목을 클릭했을 때 결과 화면이 다시 열리고 보고서 파일을 받을 수 있으면 통과. 새 분석 실행을 한 번도 하지 않아도 된다.

**Acceptance Scenarios**:

1. **Given** 과거에 완료된 분석 작업이 3건 있다, **When** 사용자가 이력 화면에 진입한다, **Then** 3건이 최신순으로 표시되며 각 행에 학과·교수명·과목명·기간·생성일시·상태(완료/실패/진행중/중단됨)가 보인다.
2. **Given** 이력 목록이 표시되어 있다, **When** 사용자가 완료된 분석 항목을 클릭한다, **Then** 해당 분석의 결과 화면이 열리고 보고서 다운로드 링크가 동작한다.
3. **Given** 이력 항목 중 하나가 진행 중 상태다, **When** 사용자가 그 항목을 클릭한다, **Then** 진행률 화면(User Story 1의 진행 화면)이 열려 현재 단계가 표시된다.
4. **Given** 재사용 탐지에서 중복 의심 영상이 검출된 분석이 있다, **When** 사용자가 그 결과 화면에서 의심 영상에 대해 [중복 확정] 또는 [오탐] 리뷰 상태를 변경한다, **Then** 변경 결과가 영속 저장되어 다음 분석부터 동일 영상 쌍이 재경고되지 않는다(spec 007 FR-013과 일치).

### User Story 3 - 운영자가 신규 학과 채널을 1회 등록하면 사용자가 즉시 사용한다 (Priority: P3)

DX지원센터장(운영자)이 신규 학과의 OAuth 클라이언트 시크릿·API 키를 agenix에 등록하고 학과↔환경변수명 매핑을 시스템에 반영한 뒤 OAuth 동의를 1회 완료하면, 사용자는 다음 새로고침에서 해당 학과를 드롭다운에서 보고 즉시 분석을 시작할 수 있다. 토큰 갱신·API 키 관리는 운영자에게만 노출되고, 사용자는 학과 이름만 본다.

**Why this priority**: P1·P2가 동작해도 운영자가 매번 시크릿을 손봐야 한다면 운영 부담이 사라지지 않는다. 다만 운영자 셋업은 학과 신규 추가 시(빈도 낮음)에만 발생하므로 우선순위는 P3.

**Independent Test**: 운영자가 신규 학과 `nursing`을 등록(agenix 시크릿 추가 + 매핑 yaml 업데이트 + OAuth 동의 완료)한 후, 사용자 화면 새로고침 시 학과 드롭다운에 `간호학과`가 추가되고 추가 인증 없이 분석이 시작되면 통과.

**Acceptance Scenarios**:

1. **Given** 운영자가 agenix에 신규 학과 `nursing`의 OAuth 시크릿·API 키를 등록하고 학과 매핑(`nursing` → 표시명 "간호학과")을 시스템에 반영한 후 OAuth 동의 흐름을 완료했다, **When** 사용자가 분석 화면을 새로고침한다, **Then** 학과 드롭다운에 "간호학과"가 추가되어 보이고 선택 가능하다.
2. **Given** 등록된 학과의 OAuth 토큰이 만료되어 분석이 실패했다, **When** 사용자가 그 학과로 분석을 시도한다, **Then** "운영자에게 인증 갱신을 요청하세요"라는 안내가 표시되고, 운영자에게는 CLI 상태 명령(예: `tube-scout admin status`) 또는 로그를 통해 갱신 필요 사실이 노출된다.

---

### Edge Cases

- **OAuth 토큰 만료/무효**: 분석 시작 직후 토큰 만료가 감지되면 즉시 실패 처리 후 사용자에게는 "운영자 문의" 한국어 메시지, 운영자에게는 갱신 필요 표시.
- **YouTube API 쿼터 초과**: 일일 쿼터를 초과한 학과로 분석 시 "오늘 API 쿼터 초과 — 내일 다시 시도하거나 운영자에게 문의" 메시지 노출.
- **자막이 0건인 영상이 다수**: 비공개 자막 비율이 매우 높을 경우(memory 등록 사실: 비공개 88.6%), force-ssl scope로 OAuth 자막 API 시도. 둘 다 실패한 영상은 "자막 없음"으로 표시되고 재사용 탐지에서 제외됨.
- **댓글 수집 비활성화**: 부산보건대 정책으로 강의 영상 댓글이 비활성화되어 있어 댓글 수집은 시도하지 않음(데이터 없음으로 처리).
- **매칭 영상 0건**: 빈 보고서 대신 "조건에 맞는 영상이 없습니다" 메시지 표시.
- **분석 도중 서버 재시작**: 진행 중이던 작업은 "중단됨" 상태로 이력에 남고, 사용자가 [재실행] 시 spec 007 FR-004의 checkpoint를 활용해 마지막 성공 단계 다음부터 이어 실행한다(처음부터 재수집 금지). 부분 산출물은 디스크에 보존되지만 UI에는 노출하지 않는다.
- **브라우저 탭 종료 후 재접속**: 진행 중 작업은 이력에서 진행 중 상태로 보이며, 클릭 시 진행률 화면이 다시 열림(작업은 서버에서 계속 실행).
- **잘못된 입력**: 빈 학과/교수명/과목명, 끝일 < 시작일, 미래 기간 등은 시작 전 폼 검증으로 한국어 오류 표시.
- **결과 파일 손상/삭제**: 이력에서 클릭했는데 보고서 파일이 디스크에서 사라진 경우 "파일을 찾을 수 없습니다 — 재실행이 필요합니다" + [재실행] 버튼 제공.
- **동일 학과 동시 분석 요청**: 사용자가 1명이라 일반적으로 발생하지 않으나, 의도치 않은 두 번째 시작 클릭 또는 재로드는 "동일 학과 분석이 이미 진행 중입니다 — 잠시 후 다시 시도하세요"로 거부.

## Requirements *(mandatory)*

### Functional Requirements

#### 인증 및 접근 제어 (User Story 1)

- **FR-001**: 시스템은 지정된 아이디·비밀번호 1쌍을 보유한 사용자만 웹 UI에 접근하도록 로그인 화면을 제공해야 한다.
- **FR-002**: 시스템은 미인증 상태에서 분석·이력·운영자 화면 등 모든 기능 화면에 접근하면 로그인 화면으로 강제 이동시켜야 한다.
- **FR-003**: 시스템은 로그인 자격 증명을 agenix를 통해 관리되는 시크릿(환경변수 참조)에서 읽어야 하며, 평문 yaml 또는 코드베이스에 비밀번호를 저장해서는 안 된다.
- **FR-004**: 시스템은 로그인 실패·세션 만료 등 인증 오류를 한국어 메시지로 표시해야 하며, 환경변수명·내부 경로는 노출하지 않아야 한다.
- **FR-004a**: 시스템은 로그인 성공 후 세션을 발급하고, 마지막 활동 기준 8시간이 경과하면 세션을 자동 만료시켜 다음 요청에서 로그인 화면으로 이동시켜야 한다.
- **FR-004b**: 시스템은 모든 사용자 트래픽을 HTTPS로 처리해야 하며(리버스 프록시 종단 또는 직접 TLS), HTTP 접근은 거부하거나 HTTPS로 리다이렉트해야 한다.
- **FR-004c**: 시스템은 동일 아이디에 대해 5회 연속 로그인 실패가 발생하면 해당 아이디를 5분간 잠금 상태로 두고, 잠금 사유를 한국어로 안내해야 한다.
- **FR-004d**: 시스템은 세션 쿠키에 `Secure`, `HttpOnly`, `SameSite=Lax` 이상 동등 보호 속성을 적용해야 한다.

#### 학과·교수·과목·기간 입력 (User Story 1)

- **FR-005**: 시스템은 운영자가 등록한 학과 목록을 드롭다운(또는 동등한 선택 UI)으로 노출하고, 사용자가 학과명을 직접 입력할 수도 있게 해야 한다.
- **FR-006**: 시스템은 사용자가 학과·교수명·과목명·기간(시작일·종료일)을 입력하고 [분석 시작]을 누르는 흐름을 단일 폼 1단계로 제공해야 한다.
- **FR-007**: 시스템은 시작 전 폼 검증으로 (a) 빈 값, (b) 공백/특수문자만 입력, (c) 끝일 < 시작일, (d) 시작일이 미래 등의 오류를 한국어로 노출해야 한다.
- **FR-008**: 시스템은 사용자가 OAuth 토큰 파일 경로, API 키, 환경변수명, JSON 설정 파일을 직접 다루지 않아도 분석을 시작할 수 있게 해야 한다.

#### 분석 파이프라인 (v1~v4 풀 실행) (User Story 1)

- **FR-009**: 시스템은 [분석 시작] 클릭 시 학과↔OAuth alias 매핑을 통해 해당 학과의 본 채널 OAuth 자격을 자동 사용하여 다음 단계를 순차 실행해야 한다 — (1) 영상 목록 수집, (2) 메타데이터 수집, (3) 자막 수집(공개+force-ssl scope 비공개), (4) retention 수집, (5) analytics 수집, (6) 재사용 탐지(spec 007 idea4), (7) 보고서 생성(v1~v3 + 재사용 탐지).
- **FR-010**: 시스템은 사용자 입력(교수명·과목명·기간)을 기존 CLI 필터 인자(`--professor`, `--course`, `--published-after`, `--published-before`)로 변환하여 백엔드 서비스에 전달해야 한다.
- **FR-011**: 시스템은 작업을 백그라운드 비동기로 실행해야 하며, 사용자가 화면을 닫아도 작업은 서버에서 계속 진행되어야 한다.
- **FR-012**: 시스템은 부산보건대 강의 영상 댓글 비활성화 정책에 따라 댓글 수집을 시도하지 않거나, 시도하더라도 "비활성화"로 일관되게 처리해야 한다.

#### 진행률 표시 (User Story 1)

- **FR-013**: 시스템은 분석 작업의 현재 단계와 단계별 처리 카운트를 5초 이내 간격으로 사용자 화면에 갱신해야 한다.
- **FR-014**: 시스템은 작업이 정상 완료되면 결과 화면으로 자동 전환하거나 결과 화면 링크를 활성화해야 한다.
- **FR-015**: 시스템은 작업 중 발생한 모든 오류를 비개발자가 이해할 수 있는 한국어 메시지로 변환하여 표시해야 하며, agenix 키 이름·환경변수명·스택 트레이스·내부 경로를 노출해서는 안 된다.

#### 보고서 접근 (User Story 1, 2)

- **FR-016**: 시스템은 완료된 분석에 대해 v1~v3 보고서(영상 필터·retention·analytics)와 v4 재사용 탐지 보고서(spec 007)를 모두 HTML·PDF·Excel 형식으로 같은 결과 화면에서 다운로드 또는 새 탭으로 열 수 있게 해야 한다.
- **FR-017**: 시스템은 각 분석 결과를 기존 `projects/{job-id}/...` 디렉터리 구조에 저장해야 하며, `{job-id}`는 `YYYYMMDD-HHMMSS` 형식(sub-second 충돌 시 `-2`, `-3`… 카운터 접미사)이고 Analysis Job의 작업 ID와 일치해 CLI와 웹이 동일한 결과 파일을 공유한다.
- **FR-018**: 시스템은 결과 파일이 디스크에서 사라진 경우 "파일을 찾을 수 없습니다 — 재실행 필요" 메시지와 [재실행] 진입점을 제공해야 한다.

#### 재사용 탐지 리뷰 (User Story 2 — spec 007 통합)

- **FR-019**: 시스템은 재사용 탐지 결과의 의심 영상 쌍에 대해 사용자가 [중복 확정] / [오탐] 리뷰 상태를 결과 화면 또는 이력 화면에서 변경할 수 있어야 한다(spec 007 FR-011·FR-012 위에 얹는 UI 레이어).
- **FR-020**: 시스템은 사용자가 변경한 리뷰 상태를 영속 저장하여, 다음 분석부터 같은 영상 쌍이 재경고되지 않게 해야 한다(spec 007 FR-013과 일치).

#### 이력 관리 (User Story 2)

- **FR-021**: 시스템은 모든 분석 작업(완료/실패/진행중/중단됨)을 이력으로 영속 저장하고 이력 화면에서 학과·교수명·과목명·기간·생성일시·상태를 최신순으로 노출해야 한다.
- **FR-022**: 시스템은 이력 항목 클릭 시 (a) 완료는 결과 화면, (b) 진행 중은 진행률 화면, (c) 실패/중단됨은 오류 화면 + 재실행 진입점으로 라우팅해야 한다. 실패/중단됨 항목의 부분 산출물은 UI에 다운로드 링크로 노출하지 않는다.
- **FR-022a**: 시스템은 실패/중단된 작업의 [재실행] 시 spec 007 FR-004의 checkpoint 메커니즘을 활용하여 마지막으로 성공한 단계 다음부터 이어 실행해야 한다(처음부터 전체 재수집 금지).
- **FR-023**: 시스템은 이력 데이터를 외부 DB 서버(PostgreSQL 등) 없이 파일 기반(JSON 또는 SQLite)으로 저장해야 한다.

#### 학과·시크릿 운영 (User Story 3)

- **FR-024**: 시스템은 운영자가 신규 학과를 추가하는 절차를 **CLI 명령**으로 제공해야 한다(예: `tube-scout admin add-department --alias <영문> --display <한국어 표시명> --secret-env <환경변수명>`). 운영자 전용 GUI는 제공하지 않으며, 명령은 agenix 시크릿 등록 → 학과 매핑 반영 → OAuth 동의 흐름을 안내한다.
- **FR-025**: 시스템은 운영자 셋업 완료 직후 사용자 화면 새로고침만으로 신규 학과가 드롭다운에 보이게 해야 한다(서버 재시작 없이 반영).
- **FR-026**: 시스템은 OAuth 토큰 만료가 감지되면 (a) 구조화 로그(파일 또는 journald)에 학과 alias·시각·사유를 기록하고, (b) `tube-scout admin status` 명령에서 갱신이 필요한 학과 목록을 즉시 조회 가능하게 해야 한다. 사용자에게는 "운영자 문의" 메시지로 추상화하되, 학과 alias 또는 환경변수명 등 내부 식별자는 노출하지 않아야 한다.
- **FR-027**: 시스템은 모든 학과별 시크릿(OAuth 클라이언트 시크릿, refresh token, API 키, 로그인 자격 증명)을 agenix를 통해 관리되는 환경변수로만 읽어야 하며, 코드베이스 또는 프로젝트 디렉터리 내에 평문 시크릿을 저장해서는 안 된다(사용자의 시크릿 중앙 관리 원칙 준수).

#### 동시성·격리 (User Story 1)

- **FR-028**: 시스템은 동일 학과에 대한 동시 분석 시작 요청을 거부하고 "동일 학과 분석이 이미 진행 중입니다" 메시지를 표시해야 한다.
- **FR-029**: 시스템은 서로 다른 학과에 대한 분석 작업이 동시에 실행되어도 결과 디렉터리·이력 항목이 서로 섞이지 않게 격리해야 한다.

#### 호환성 (Constraints)

- **FR-030**: 시스템은 기존 CLI 명령(`collect all`, `analyze all`, `report bundle`, spec 007의 재사용 탐지 명령)의 동작·인자·출력 위치를 변경하지 않아야 한다.
- **FR-031**: 시스템은 기존 멀티채널 OAuth 토큰 저장 위치(`~/.config/tube-scout/tokens/{alias}_token.json`)를 그대로 사용해야 한다.

### Key Entities *(데이터가 관여)*

- **Department (학과)**: 분석 대상 학과의 사용자 친화 식별자. 속성: 학과 alias(영문 식별자), 표시명(한국어), 본 채널 ID, OAuth alias, agenix 시크릿 환경변수명, 마지막 사용 시각. 운영자가 등록·관리한다. 사용자에게는 표시명만 노출.
- **Analysis Job**: 한 번의 분석 실행 단위. 속성: 작업 ID(형식: `YYYYMMDD-HHMMSS`, sub-second 충돌 시 `-2`, `-3`… 카운터 접미사), 학과 alias, 교수명, 과목명, 기간(시작일·종료일), 상태(대기/진행중/완료/실패/중단됨), 시작·완료 시각, 현재 단계(7단계 중), 단계별 처리 카운트, 결과 디렉터리 경로(서버 내부, 디렉터리명 = 작업 ID), 오류 메시지(실패 시).
- **Analysis Result**: 완료된 작업의 산출물 묶음. 속성: 작업 ID(외래키), v1~v3 보고서 경로(HTML/PDF/Excel), 재사용 탐지 보고서 경로(HTML/Excel), 매칭 영상 수, 의심 영상 쌍 수(우선순위별 카운트). 기존 `projects/YYYYMMDD-HHMMSS/...` 구조와 1:1 매칭.
- **Review Status**: 재사용 탐지 결과 영상 쌍의 사용자 검토 상태. 속성: 영상 쌍 ID, 상태(미검토/중복확정/오탐), 변경 시각, 변경자. spec 007 FR-011~013과 동일 모델.
- **Operator Action**: 운영자 동작 로그(학과 등록, OAuth 동의 완료, 토큰 갱신 등). 속성: 동작 종류, 대상 학과 alias, 운영자 식별자, 시각.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: **진입 단순성** — 비개발자 테스터가 사전 교육 없이 로그인 후 단일 입력 폼에서 학과·교수명·과목명·기간 4개 입력 + [시작] 클릭으로 분석 시작에 도달할 수 있다(테스트: 사용성 검증 5명 중 5명 5분 이내 성공).
- **SC-002**: **셋업 부담 제로** — 사용자는 환경변수, agenix 키 이름, 토큰 파일, JSON/yaml 설정을 한 번도 편집하지 않고 분석을 시작·완료·재열람할 수 있다(테스트: 비개발자 사용 도중 터미널/파일시스템 접근 0회).
- **SC-003**: **진행률 가시성** — 분석 진행 중 화면이 5초 이내 간격으로 단계 또는 처리 카운트를 갱신하며, "끝났는지 모르겠다"는 사용자 응답이 사용성 검증에서 0건이다.
- **SC-004**: **보고서 즉시 접근** — 분석 완료 후 같은 화면에서 클릭 1회로 v1~v3 보고서 + 재사용 탐지 보고서의 HTML/PDF/Excel 모든 산출물에 도달할 수 있다.
- **SC-005**: **이력 재열람 효율** — 과거 결과를 다시 열기까지 [이력 진입 → 항목 클릭 → 보고서 열림]이 평균 10초 이내(클릭 2회 이하)로 완료된다.
- **SC-006**: **시크릿 비노출** — 사용자는 OAuth 토큰·API 키·agenix 환경변수명·채널 ID를 화면에서 보거나 입력하지 않는다(검증: UI 화면 캡처에 시크릿 정보 0건 노출).
- **SC-007**: **운영자 1회 셋업** — 운영자가 신규 학과 1개를 등록(agenix 시크릿 추가 + 매핑 반영 + OAuth 동의)하기까지 소요 시간 10분 이하, 이후 사용자 추가 인증 0회.
- **SC-008**: **CLI 동등 결과** — 동일 학과·교수·과목·기간으로 (a) CLI와 (b) 웹 UI를 각각 실행했을 때 산출되는 보고서 매칭 영상 수, retention/analytics 수치, 재사용 탐지 의심 영상 쌍 수가 일치한다(차이 0건).
- **SC-009**: **재사용 탐지 리뷰 영속성** — 사용자가 영상 쌍에 [중복 확정] / [오탐]을 지정하면 다음 분석부터 동일 쌍이 재경고되지 않는다(spec 007 FR-013 검증과 동일).
- **SC-010**: **풀 파이프라인 성공률** — 정상 입력 + 정상 OAuth/API 환경에서 7단계 파이프라인(목록→메타→자막→retention→analytics→재사용 탐지→보고서)이 끝까지 실행되어 보고서가 생성되는 비율 ≥ 95%(샘플 20회 기준).
- **SC-011**: **인증 보호 베이스라인** — (a) 마지막 활동 기준 8시간 후 세션 만료가 동작, (b) HTTP 접근이 거부 또는 HTTPS로 리다이렉트, (c) 5회 실패 후 5분 잠금이 동작하는 것이 모두 확인된다(검증: 통합 테스트 3건).

## Assumptions

- **사용자 수**: 사용자는 부산보건대 교무과 단일 담당자 1명이며, 동시 분석 작업은 1건이 일반적이다(다중 사용자 시나리오는 본 idea 범위 외).
- **운영 환경**: 사내망(부산보건대 캠퍼스망) 또는 운영자가 통제하는 단일 서버에서 구동되며, 인터넷 외부 노출은 가정하지 않는다. 외부 노출 필요 시 별도 보안 검토.
- **시크릿 저장**: 모든 OAuth 클라이언트 시크릿, refresh token, API 키, 로그인 자격 증명은 agenix 중앙 저장소에 보관하고 환경변수로 참조한다. 코드베이스·프로젝트 디렉터리에 평문 시크릿이 저장되지 않는다(사용자 정책).
- **본 채널 OAuth**: 분석 대상은 학과 본 채널이며 운영자가 OAuth 동의를 완료하므로 retention/analytics/비공개 자막 등 비공개 통계까지 풀 수집이 가능하다.
- **댓글**: 부산보건대 강의 영상은 학교 정책으로 댓글이 비활성화되어 있어 댓글 데이터는 수집하지 않는다(0건이 정상 상태).
- **재사용 탐지 의존성**: idea4(spec 007)의 재사용 탐지 기능이 본 idea의 보고서 범위에 포함된다. spec 007 FR-001~020을 그대로 호출하며, 본 idea는 그 위에 웹 UI 레이어만 추가한다.
- **알림**: 분석 완료 알림은 화면 폴링/자동 갱신만으로 충분하다(이메일/슬랙은 본 idea 범위 외).
- **이력 보존**: 분석 결과는 무기한 보존하며 디스크 용량 관리는 운영자가 수동으로 수행한다(자동 삭제 정책은 범위 외).
- **진행률 표시 수준**: 단계명 + 단계별 처리 카운트(예: "자막 수집 12 / 47 영상")로 충분하며, 영상별 100% 정밀 백분율은 요구하지 않는다.
- **언어**: UI와 모든 사용자 메시지는 한국어 전용이다.
- **백엔드 재사용**: 분석 파이프라인은 기존 서비스 계층(`VideoFilterService`, `VideoReportGenerator`, `YouTubeDataService`, `collect all`/`analyze all`/`report bundle` 통합 명령, spec 007 재사용 탐지 명령)을 함수 호출로 재사용하며, 신규 분석 알고리즘은 추가하지 않는다.
- **저장소**: 작업/이력 메타데이터는 외부 DB 없이 파일 기반(JSON 또는 SQLite)으로 저장하며, 분석 결과 파일은 기존 `projects/YYYYMMDD-HHMMSS/...` 구조를 따른다.
- **OAuth 갱신**: refresh token 자동 순환은 spec 005에서 구현된 메커니즘을 그대로 사용하며, 갱신 실패 시에만 운영자 수동 개입이 필요하다.
- **범위 한정**: 본 idea는 v1~v4 기능을 비개발자 1명이 쓸 수 있게 하는 데 한정한다. 새로운 분석 기능, 다국어 UI, 모바일 전용 UI, 다중 사용자, 외부 사용자 셀프서비스 회원가입은 모두 범위 외.

## Dependencies

- 기존 CLI 통합 명령(`collect all`, `analyze all`, `report bundle`)이 안정적으로 동작해야 한다(spec 003·004·006).
- 멀티채널 OAuth alias 토큰 관리(`~/.config/tube-scout/tokens/{alias}_token.json`, force-ssl scope, spec 003·005)가 동작해야 한다.
- 보고서 생성기(`VideoFilterService`, `VideoReportGenerator`, spec 004·006)가 PDF/HTML/Excel 산출물을 생성할 수 있어야 한다.
- **재사용 탐지(spec 007 — idea4)**가 구현 완료되어 자막 수집 + 5개 지표 계산 + 의심도 점수 + 리뷰 상태 관리를 제공해야 한다. 본 idea는 그 위에 웹 UI 레이어만 추가하므로 spec 007 미완료 시 기능 범위가 v1~v3로 축소된다.
- agenix 중앙 저장소가 운영 환경에 셋업되어 있고 NixOS devShell이 환경변수 주입을 지원해야 한다.

## Out of Scope

- 새로운 분석 알고리즘, 신규 데이터 소스 추가
- 댓글 데이터 분석(학교 정책으로 비활성화 상태)
- 다중 사용자 동시 운영(현재 사용자 1명 가정)
- 모바일 전용 UI 또는 반응형 우선 설계(데스크톱 브라우저 우선)
- 다국어 UI(한국어 전용)
- 외부 사용자(학교 외) 대상 셀프서비스 회원가입
- 이메일/슬랙 등 외부 알림 채널 연동
- 자동 결과 보존/삭제 정책(수동 운영으로 시작)
- 분석 결과 시각화 신규 차트(기존 보고서 그대로 재사용)
- agenix 시크릿 등록을 GUI로 자동화(운영자는 기존 NixOS/agenix 흐름 사용)
