# spec 016 입력 — Takeout 적재 결함 보고서 (간호학과 9개 샘플 검증)

**작성일**: 2026-05-15
**작성 맥락**: 사용자 결정으로 분석 작업 흐름 전체를 Takeout 기반으로 확정한 뒤, master 의 `v0.5.0` 코드가 실제 Takeout 결과물에서 작동하는지 확인하려고 간호학과 샘플 9개 영상으로 quickstart 절차를 처음부터 시도하다가 발견된 결함 묶음. 본 문서는 spec 016 (Takeout 기반 분석 기반 구축) 의 사양 작성 시 첫 번째 입력으로 들어간다.

**처리 정책 (사용자 결정 2026-05-15)**: 결함은 본 문서에 기록만 한다. 수정 작업은 spec 016 작성에 들어가는 시점에 일괄 진행한다.

**입력 데이터**:
- 경로: `data/takeout-20260511T130817Z-3-001/Takeout/`
- 분량: 간호학과 채널 9개 영상, 약 9.9 기가바이트 (간호학과 전체는 약 2.4 테라바이트)
- 비고: 간호학과 한 채널만 2.4 테라바이트이므로 부산보건대학교 22 학과 전체는 수십 테라바이트 규모로 추정됨. 모든 영상의 본격 분석은 별도의 GPU 서버에서 수행되며, 본 작업 머신은 검증과 개발 용도로만 사용한다.

**검증 환경**: NVIDIA GeForce RTX 3060 6 기가바이트 (CTranslate2 백엔드의 CUDA 가속 정상 작동 확인됨, faster-whisper 1.2.1, CTranslate2 4.7.1 설치 완료).

## 2. 결함별 상세

### 결함 1 — 학과 등록 확인 명령이 거짓말을 한다

실행 결과:

```
$ tube-scout admin list
등록된 학과가 없습니다.
```

그러나 실제 파일을 직접 열어보면 간호학과는 정상 등록되어 있다.

```json
// ~/.config/tube-scout/tokens/channels.json
{
  "nursing": {
    "alias": "nursing",
    "channel_id": "UCnh3tm9uQkyA260cAHfl9rg",
    "channel_name": "부산보건대 간호학과",
    "registered_at": "2026-05-08T14:15:55.990557+00:00",
    ...
  }
}
```

원인은 등록부가 두 곳으로 갈라져 있기 때문이다.

- 한 곳: `~/.config/tube-scout/tokens/channels.json` — 사양 003에서 정의된 OAuth 등록부. `auth.load_registry()` 가 이쪽을 본다. Takeout 적재 명령 `collect takeout` 도 이쪽을 본다.
- 다른 곳: `departments.json` — 사양 008(웹 관리 화면) 에서 추가된 학과 메타 저장소. `admin list` 와 `admin add-department` 는 이쪽만 본다.

같은 "간호학과" 가 한쪽에는 있고 다른 쪽에는 없는 상태이므로, 운영자 입장에서는 등록을 두 번 해야 한다. 두 등록부를 하나로 통일하거나, 적어도 `admin list` 가 두 곳을 모두 합쳐 보여주어야 한다.

분석 자체는 `collect takeout` 이 첫 번째 등록부를 읽기 때문에 진행 가능하므로, 이 결함은 사용자 체감 혼란에 그친다. 그러나 운영자에게 "등록이 안 되어 있다" 는 거짓 정보를 주는 것이므로 시간 낭비와 혼동 비용이 크다.

### 결함 2 — 학과 등록 명령이 OAuth 환경변수를 강제한다

`tube-scout admin add-department` 의 옵션 정의는 다음과 같다.

- `--channel-id-env`: agenix 환경변수명 (필수)
- `--client-secret-env`: agenix 환경변수명 (필수)
- `--api-key-env`: agenix 환경변수명 (필수)

그리고 내부적으로 `_check_envs_present()` 가 위 세 환경변수가 실제로 존재하는지까지 확인한 뒤 없으면 종료 코드 1로 끝난다.

이 세 환경변수는 모두 YouTube API 호출(OAuth) 용도다. 그런데 2026-05-12 결정으로 우리는 YouTube API 호출 경로를 영구히 폐기하고 Google Takeout 으로만 데이터를 받기로 했다. 즉 신규 학과 등록 단계에서부터 사용하지 않을 OAuth 자격 증명을 강제로 받고 있다.

해결 방향은 크게 두 갈래다. (a) `add-department` 에 `--takeout-only` 옵션을 추가해 OAuth 자격 검증을 건너뛰게 만든다. (b) Takeout 전용 신규 등록 명령을 별도로 만든다. spec 016 작성 시 둘 중 어느 쪽이 사용자 체감과 코드 응집에 더 맞는지 결정 필요.

### 결함 3 — `채널.csv` 컬럼명 가정 불일치 (최우선 차단)

실행 결과:

```
$ tube-scout collect takeout --takeout-dir data/takeout-20260511T130817Z-3-001/Takeout --channel nursing --dry-run
Alias error: Missing columns in 채널.csv: {'채널 이름'}
```

코드(`src/tube_scout/services/takeout_ingest.py:49`)는 다음 두 컬럼을 필수로 요구한다.

```python
_CHANNEL_CSV_REQUIRED = {"채널 ID", "채널 이름"}
```

그런데 실제 Google Takeout 이 만들어 준 `채널.csv` 의 컬럼 헤더는 다음과 같다.

```
채널 ID, 채널 국가, 채널 태그 1, 채널 제목(원본), 채널 공개 상태
```

"채널 이름" 이라는 이름의 컬럼은 아예 존재하지 않는다. 비슷한 정보는 "채널 제목(원본)" 컬럼에 들어 있다. 다른 코드 위치(`_parse_channel_csv()`) 에서도 `row["채널 이름"]`, `row.get("국가", "")` 같은 잘못된 컬럼 참조가 이어진다.

이것은 단순한 오탈자 한 줄이 아니다. **사양 013 의 Takeout 적재 코드를 작성할 때 실제 Google Takeout 결과물을 한 번도 열어보지 않고 "컬럼명이 이렇겠지" 하고 추측해서 작성했다**는 사실을 의미한다. 다음 결함 4 가 같은 사실을 한 번 더 확증한다.

### 결함 4 — `동영상.csv` 컬럼명 가정 불일치 (최우선 차단)

코드(`src/tube_scout/services/takeout_ingest.py:45`)가 영상 메타데이터 파일에서 요구하는 컬럼은 다음 9개다.

```python
_VIDEO_CSV_REQUIRED = {
    "동영상 ID", "동영상 제목", "동영상 URL", "동영상 생성 타임스탬프",
    "근사치 길이(밀리초)", "채널 ID", "카테고리", "공개상태", "오디오 언어",
}
```

실제 `동영상(1).csv` 의 컬럼 헤더는 다음과 같다.

```
동영상 ID, 근사치 길이(밀리초), 동영상 오디오 언어, 동영상 카테고리,
동영상 설명(원본) 언어, 채널 ID, 동영상 제목(원본), 동영상 제목(원본) 언어,
개인 정보 보호, 동영상 상태, 동영상 생성 타임스탬프
```

대조표는 다음과 같다.

| 코드가 요구하는 컬럼 | 실제 컬럼 | 상태 |
|---|---|---|
| 동영상 ID | 동영상 ID | 일치 |
| 동영상 제목 | 동영상 제목(원본) | 이름 다름 |
| 동영상 URL | (없음) | **컬럼 자체가 존재하지 않음** |
| 동영상 생성 타임스탬프 | 동영상 생성 타임스탬프 | 일치 |
| 근사치 길이(밀리초) | 근사치 길이(밀리초) | 일치 |
| 채널 ID | 채널 ID | 일치 |
| 카테고리 | 동영상 카테고리 | 이름 다름 |
| 공개상태 | 개인 정보 보호 또는 동영상 상태 | 이름 다름 — 둘 중 어느 쪽이 의미상 맞는지 확인 필요 |
| 오디오 언어 | 동영상 오디오 언어 | 이름 다름 |

특히 `동영상 URL` 컬럼은 실제로 존재하지 않는다. Google Takeout 은 영상 식별자(video_id) 만 제공하며 URL 은 식별자에서 일반 규칙으로 도출해야 한다(`https://youtu.be/<video_id>` 형태).

### 결함 5 — 분할 영상 메타데이터 파일 (정보 등급)

간호학과 한 채널의 영상 메타데이터 파일이 한 archive 안에서 다음과 같이 10개로 분할되어 있다.

```
동영상(1).csv, 동영상(2).csv, ... , 동영상(10).csv
```

코드의 `meta_dir.glob("동영상*.csv")` 는 위 패턴을 잡아주므로 동작에는 문제가 없다. 다만 quickstart §1.2 "분할 Takeout part 누적" 절은 archive 단위 분할만 다루고 있어, **한 archive 안에서도 메타데이터 파일이 분할될 수 있다는 사실**이 문서에 명시되어 있지 않다. spec 016 작성 시 이 분할이 항상 발생하는지, 분할 단위가 무엇인지 (영상 수? 파일 크기?) 확인 후 문서화 필요.

## 3. 구조적 의미

결함 3 과 결함 4 는 단일 결함이 아니라 **사양 013 의 Takeout 적재 모듈 전반이 추정 위에 놓여 있다는 사실**을 드러낸다.

- `src/tube_scout/services/takeout_ingest.py` 가 실제 Google Takeout 결과물의 형식을 한 번도 직접 확인하지 않은 상태에서 작성됐다.
- 컬럼명을 정의하는 상수 두 개와 `_parse_channel_csv()`, 영상 메타데이터 파싱 본문이 모두 추정 컬럼명에 의존한다.
- master 의 `v0.5.0` 상태는 사실상 "Takeout 실데이터로 한 번도 작동시킨 적이 없는" 상태다.

따라서 spec 016 의 가장 첫 번째 작업 묶음은 단순한 컬럼 이름 수정이 아니라 **Takeout 결과물의 실제 구조를 모두 실측한 뒤 적재 모듈을 처음부터 다시 작성하는 일**이 된다. 이때 함께 확인해야 할 항목은 다음과 같다.

- mp4 영상 파일이 들어 있는 폴더 이름이 코드 가정(`_VIDEO_SUBDIR`)과 일치하는가
- 자막 파일이 Takeout 안에 동봉되는가 (자막 파일 형식, 어느 폴더에 있는지)
- "동영상 메타데이터" 가 아닌 다른 폴더(시청 기록, 댓글, 채널, 재생목록 등) 안에 분석에 활용 가능한 정보가 있는가
- 영상 메타데이터의 분할 단위가 무엇인지

## 5. 참고 — 환경 확인 결과

본 작업 진행 중 부수적으로 다음 사실이 확인되었으므로 spec 016 작성 시 참고하면 좋다.

- 본 머신의 GPU 환경은 정상이다. `nvidia-smi` 가 NVIDIA GeForce RTX 3060 (메모리 6 기가바이트, 드라이버 595.71.05) 을 인식하며, CTranslate2 백엔드를 통한 faster-whisper 의 CUDA 모드 모델 로딩이 1초 안에 성공한다. PyTorch 는 설치되지 않았지만 faster-whisper 는 PyTorch 의존이 없는 설계이므로 이상 없다.
- `uv sync --extra asr` 한 줄로 faster-whisper 1.2.1 과 CTranslate2 4.7.1 이 설치된다. 다만 이 명령은 `dev` 추가 묶음을 떨어뜨리므로 검증 단계에서 pytest 등을 함께 쓰려면 `uv sync --extra asr --extra dev` 로 두 묶음을 동시에 동기화해야 한다.
- RTX 3060 의 메모리 6 기가바이트는 medium 크기까지의 모델에 안전하다. large-v3 모델은 메모리 한계에 가까워지므로 본 머신에서는 권장하지 않으며, 별도 GPU 서버 단계에서 사용한다.
