# Tube Scout v5 — 행정 담당자용 간편 웹 UI

## Background

Tube Scout는 v1~v4를 거치며 강의 영상 수집·분석·재사용 탐지·보고서 생성까지 end-to-end 파이프라인을 갖추었다. 그러나 현재 진입점은 모두 CLI(터미널 명령) 뿐이고, 환경변수와 OAuth 토큰을 직접 다뤄야 동작한다.

실사용자는 부산보건대학교 학과 사무실의 행정 담당자다. 이들은 비개발자이며, 터미널·DB·환경변수·JSON 편집에 익숙하지 않다. 한 번 분석을 돌리려면 개발자(DX지원센터장)에게 의뢰하거나 매번 명령을 받아 적어 실행해야 하는 상황이다.

별도로 시도된 ytsubs 프로토타입은 FastAPI + PostgreSQL + ChromaDB 기반이라 행정 담당자가 운영하기에 더 복잡한 구조였고, 핵심 알고리즘은 이미 Tube Scout v4(idea4)에 통합되어 별도 유지의 실익이 사라졌다.

## Problem

- **행정 담당자**가 분석을 직접 실행할 수 없어, 매번 개발자에게 요청을 거쳐야 하므로 보고서 생성 리드타임이 길어진다
- 환경변수(`TUBE_SCOUT_CLIENT_SECRET`), OAuth 토큰 파일 경로, JSON 설정 편집 등 진입 장벽이 높다
- 분석에 수 분 ~ 수십 분이 걸리는데 진행 상황이 터미널 로그로만 노출되어, 비개발자에게는 "끝났는지" 판단이 어렵다
- 과거 생성한 보고서를 다시 찾으려면 `projects/YYYYMMDD-HHMMSS/03_report/...` 경로를 직접 탐색해야 한다

## Desired Outcome

행정 담당자가 사내 웹 주소에 접속하여 채널을 고르고 검색 키워드를 입력하면, 별도의 셋업이나 명령어 없이 분석이 실행되고, 완료 후 보고서를 다운로드할 수 있다. 과거 분석 이력도 같은 화면에서 다시 열람할 수 있다.

### User Scenarios

- **학과 행정 담당자**는 터미널을 열지 않고 웹페이지에서 채널 alias를 고르고 "감염미생물학" 같은 키워드만 입력하면 분석이 시작되기를 원한다. 왜냐하면 명령어와 옵션을 외울 필요 없이 일상 업무 도구처럼 쓰고 싶기 때문이다.
- **학과 행정 담당자**는 분석 진행률이 화면에 표시되고, 끝나면 알림과 함께 보고서 다운로드 링크가 뜨기를 원한다. 왜냐하면 수십 분이 걸리는 작업을 마냥 기다리거나 새로고침으로 확인하는 것은 비효율적이기 때문이다.
- **학과 행정 담당자**는 채널별로 다른 API 키/OAuth 인증을 신경 쓰지 않고 alias만 고르고 싶다. 왜냐하면 키 관리는 운영자(DX지원센터)가 하고, 사용자는 어떤 채널을 분석할지만 정하면 되기 때문이다.
- **DX지원센터장**은 행정 담당자가 과거에 만든 보고서를 다시 열어 학과장 보고에 재활용하고 싶다. 왜냐하면 동일한 분석을 매번 다시 돌리는 것은 시간과 API 쿼터 낭비이기 때문이다.
- **DX지원센터장**은 신규 채널 알리아스를 1회 등록하면, 이후 행정 담당자가 별도 인증 절차 없이 그 채널을 선택할 수 있기를 원한다. 왜냐하면 OAuth 흐름은 운영자만 한 번 진행하면 충분하기 때문이다.

## Constraints

- 외부 DB 서버(PostgreSQL 등)에 의존하지 않아야 한다 — 행정 담당자가 운영하는 환경에서 DB 셋업/관리 부담을 만들지 않는다
- 기존 CLI 명령(`collect all`, `analyze all`, `report bundle`)의 동작을 변경하지 않아야 한다 — 웹 레이어는 기존 기능 위에 얇게 얹는다
- 멀티채널 OAuth 토큰 저장 위치(`~/.config/tube-scout/tokens/{alias}_token.json`)는 기존 구조를 그대로 사용한다
- 비개발자 운영을 전제로 하므로, 사용 중 발생하는 오류는 사용자가 이해할 수 있는 한국어 메시지로 표시되어야 한다
- 분석 결과 파일(JSON/Parquet/HTML)은 기존 `projects/타임스탬프/...` 구조를 보존하여 CLI와 웹이 같은 데이터를 공유해야 한다

## Success Criteria

- SC-1: 진입 단순성 — 행정 담당자가 별도 교육 없이 웹페이지 첫 화면에서 분석을 시작할 수 있다 (채널 선택 + 키워드 입력 + 시작 버튼, 3단계 이내)
- SC-2: 셋업 부담 제로 — 사용자는 환경변수, 토큰 파일, JSON 설정을 편집하지 않는다 (운영자 1회 셋업으로 충분)
- SC-3: 진행률 가시성 — 분석 단계(수집/분석/보고서 생성)와 백분율 또는 단계별 상태가 실시간으로 화면에 표시된다
- SC-4: 보고서 즉시 접근 — 분석 완료 후 같은 화면에서 HTML/PDF/Excel 보고서를 다운로드 또는 새 탭으로 열 수 있다
- SC-5: 이력 재열람 — 과거 분석 결과 목록(채널·키워드·생성일시)을 보고 클릭만으로 보고서를 다시 열 수 있다
- SC-6: 채널 자동 매칭 — 사용자가 채널 alias를 선택하면 해당 OAuth 토큰이 서버에서 자동으로 사용된다 (사용자가 키 정보를 입력하거나 보지 않는다)
- SC-7: 동시 사용 — 여러 행정 담당자가 동시에 서로 다른 채널을 분석해도 결과가 섞이지 않는다

## Discussion Notes

> 본 idea는 ytsubs(LVDDS) 프로토타입의 운영 부담 문제에서 출발했다. ytsubs는 FastAPI + PostgreSQL + ChromaDB로 동일한 기능 일부를 시도했으나, 행정 담당자에게 PG 클러스터 초기화/기동/마이그레이션을 요구하는 구조라 실사용 진입 장벽이 높았다. 핵심 알고리즘 4가지(이중 탐지, Processing Status, 관리자 리뷰, Whisper fallback)는 이미 Tube Scout idea4(spec 007)에 통합 완료되어 ytsubs 자체는 정리하기로 결정했다.
>
> Tube Scout 면밀 조사 결과, 본 UI 추가에 필요한 백엔드 요소는 모두 이미 존재한다 — `collect all`/`analyze all`/`report bundle` 통합 명령, `--keyword`/`--professor`/`--course`/`--published-after/before` 필터링, 멀티채널 OAuth alias 토큰 관리, 외부 DB 비의존(JSON/Parquet/SQLite). 빠진 것은 "얇은 웹 레이어 한 층"뿐이다.
>
> 작업 규모는 1~2주(1인 풀타임)로 추정된다. 백엔드 로직 신규 작성 없이 기존 서비스 계층(`VideoFilterService`, `VideoReportGenerator`, `YouTubeDataService` 등)을 함수 호출로 재사용하고, 폼·진행률·이력 UI만 신규 구현하는 범위다.
>
> 본 idea의 범위는 "v1~v4 기능을 행정 담당자가 쓸 수 있게 하는 것"에 한정한다. 새로운 분석 기능 추가는 별도 idea로 분리한다.
